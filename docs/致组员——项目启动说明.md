# 致组员——项目启动说明

## 一、我们要做什么

**南京大学智能校园地图网站**。一句话说：能搜地点、能在地图上看到路线、点进建筑能看到每层楼的走廊和房间、告诉你从A到B怎么走（在外面走哪条路、进了楼走哪个楼梯上几楼）。

面向的是南京大学仙林校区。初期检查（半个月后）交三样东西：**技术文档 + PPT + 核心代码**。最终（一个月后）交完整项目。

---

## 二、我们怎么做的决策

下面是我们讨论后达成的一致意见，知道"为什么"比知道"是什么"重要：

### 技术方面

| 决定 | 原因 |
|------|------|
| 初期只用 HTML/CSS/JS，不加框架 | 先做出能跑的东西应付初期检查，框架学习成本太高 |
| 但地图用了 Leaflet.js 库 | 纯手写地图两周根本做不出来，Leaflet 小而精，不绑定框架 |
| 室内地图用 Canvas 画 | 室内不像室外有现成地图瓦片，得自己画走廊房间，Canvas 最灵活 |
| 路径规划用 A* 算法自己写 | 算法本身不难，自写可控，后期优化也方便 |
| 数据全用 JSON 文件 | 不用数据库，不用后端，静态文件就能跑 |
| 初期检查后再考虑重构 | 先让东西能跑，后面有时间再引入 React/Vue 等现代框架 |

### 协作方面

| 决定 | 原因 |
|------|------|
| GitHub 协作 | 免费，业界标准，老师也认可 |
| 每人一个开发分支 | 互不干扰，出问题不影响别人 |
| PR 合并到 main | 有人把关，避免乱合并 |
| 本地演示（不部署服务器） | 初期检查不需要线上部署，省事 |

### 🅱️ 路径规划（nav/）

**你负责"怎么找到路"**

具体要做的：
- 实现 Graph 类（节点、边、邻接查询）—— 已写好骨架在 `js/nav/graph.js`
- 实现 A* 寻路算法 —— 已写好骨架在 `js/nav/astar.js`

- 把建筑入口做成"桥梁节点"，连接室外图和室内图
- 把路径结果分段（室外段 / 2F段 / 楼梯 / 3F段），方便渲染和显示
- 室外边权优先使用 `outdoor-paths.json` 的折线长度，而不是两个节点间的直线距离
- 维护 `npm.cmd run validate:data`，让数据问题在联调前暴露

**你的代码在**：`js/nav/graph.js`、`js/nav/astar.js`、`js/nav/outdoor-graph.js`、`js/nav/indoor-graph.js`、`js/nav/path-renderer.js`

**核心数据结构**（必须理解）：
```js

{
  id: "yifu-2f-corridor-north",  // 唯一ID

```

## 五、路径数据怎么做

路线是否合理，关键看我们有没有把“人能走的线”标出来。不要只给建筑物一个坐标，也不要让路线从一个节点直线飞到另一个节点。

### 标注流程

1. D 找到室外地图或楼层平面图，放到 `assets/source-maps/`。
2. A 启动本地服务，打开：

   ```text
   http://localhost:8080/tools/path-editor.html
   ```

3. D 在图片上先标节点：入口、路口、走廊转角、房间门、楼梯、电梯。
4. D 再标边：沿道路中心线或走廊中心线画，不要穿墙、穿草坪、穿房间。
5. 不确定的边设为 `needs-review`。
6. 导出 JSON 后交给 B 或让 AI 辅助转换成正式 `data/outdoor-paths.json` / `data/indoor/*.json`。
7. B 运行：

   ```bash
   npm.cmd run validate:data
   ```

8. 四个人至少一起检查一条主演示路线，例如“南门 → 逸夫楼入口 → 3F A301”。

### 每个人要注意

- A：地图上显示的室外路线应沿 `edge.path` 折线走。
- B：A* 的边权应优先使用折线长度，`walkable: false` 的边不能参与寻路。
- C：路线说明要写给普通用户看，不显示节点 ID。
- D：图片看不清时不要猜，先标 `needs-review`，找更清晰图片或现场确认。

## 六、怎么开始

### 第一步：拿到代码

```bash
git clone https://github.com/xianqianqianzz/-EL_project.git
cd -EL_project
```

### 第二步：创建你的分支

```bash
# A 执行这个
git checkout -b map-render
git push -u origin map-render

# B 执行这个
git checkout -b path-planning
git push -u origin path-planning

# C 执行这个
git checkout -b ui-interaction
git push -u origin ui-interaction

# D 执行这个
git checkout -b data-docs
git push -u origin data-docs
```

### 第三步：打开你的文件，开始写

代码骨架已经搭好，每个文件里都有函数声明和注释，你只需要填充实现。**在自己的分支上开发，不要直接在 master/main 上改。**

### 第四步：每天收工时 push

```bash
git add 你改的文件
git commit -m "feat(模块): 做了什么"
git push origin 你的分支名
```

详细操作见 `docs/github-guide.md`。

- **需要新数据** → 找 D 补充
