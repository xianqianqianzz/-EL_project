# 第 4 阶段：路径修改申请与审核合并

## 目标

允许登录用户报告并标注真实路径变化，同时确保普通用户不能直接覆盖正式地图，只有工作人员审核通过后才能合并。

## 申请补丁格式

申请只允许包含：

- `add_nodes`：新增路网节点，申请内可使用临时 ID。
- `add_edges`：新增可通行边，可引用正式节点或本申请新增节点。
- `remove_edge_ids`：删除被确认错误或不可通行的正式边。

路径申请不能修改地点、底图、比例尺、区域属性或整份 `area.json`。批准时后端基于当前最新正式区域重新应用补丁，不使用用户上传的完整文件覆盖主数据。

## 用户流程

1. 在主网站登录。
2. 打开“路径反馈”，进入与主网站风格一致的路径标注工具，可随时返回校园地图。
3. 工具自动加载当前正式区域和底图，并提供滚轮缩放、拖曳平移、按钮缩放和适合窗口功能。
4. 使用添加节点、连接节点或删除错误边模式标注变化；工具不向普通用户暴露图片导入、`area.json` 导入、比例尺和 JSON 导出。
5. 填写标题和详细现场说明。
6. 点击提交时由工具隐式生成增量补丁并上传；提交不会立即改变正式地图。

## 工作人员流程

1. `staff` 或 `admin` 登录后打开“路径申请”。
2. 查看审核队列和修改数量。
3. 核验说明、现场情况和路径合理性。
4. 填写审核意见，选择批准并合并或拒绝。
5. 批准时系统重新校验当前正式地图；若申请已过期、边已变化或合并会造成断路，批准会被阻止。

## 合并安全

- 新节点和新边在批准时重新分配正式 ID，避免并发申请 ID 冲突。
- 合并后拒绝重复坐标、重复边、无效引用、孤立节点和不可达地点。
- 正式 `area.json` 使用临时文件和原子替换写入。
- 每次批准前，在用户本地应用数据目录保存正式区域备份。
- 申请、提交人、审核人、审核意见和合并摘要保存在数据库中。
- 同一服务进程内的批准操作串行执行，避免同时写入主文件。

## 当前边界

- 本阶段不会自动提交或推送 GitHub；阶段 5 再设计 GitHub 更新策略。
- 工作人员角色由数据库已有角色决定，本阶段不提供公开角色修改入口。
- 不允许通过申请修改地点；仅允许删除非地点绑定的路网节点，且必须同步删除全部关联边并通过连通性校验。

## 验收标准

1. 普通用户可提交并查看自己的路径申请，但不能审核或直接修改地图。
2. `staff/admin` 可查看队列、拒绝申请或批准并合并。
3. 重复处理、无效节点、重复边、孤立节点和断路修改会被拒绝。
4. 批准合并使用最新正式区域，自动重映射 ID，并保留审核记录和备份。
5. `npm.cmd run check` 与 `npm.cmd run db:upgrade` 通过。
6. 阶段提交已推送 GitHub `master`。

## 节点删除与地图审核

- 申请可通过 `remove_node_ids` 删除正式路网节点；标注工具删除节点时会自动同时标记其关联边。
- 地点 `routeNodeId` 绑定的节点不能通过路径反馈删除。
- 批准前，后端会确认待删除节点存在、全部关联边已删除，并再次校验剩余路网没有孤立节点或不可达地点。
- 后台路径申请页提供地图预览：原有路网、新增节点与边、删除节点与边、地点使用不同颜色叠加显示。
