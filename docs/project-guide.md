# 南京大学仙林校区地图：综合项目说明

本文档汇总项目经过多轮迭代后的有效设计、完成情况、数据规范、使用方法和部署方式。评测、维护和继续开发时应优先阅读本文；`phase-*.md` 等文件仅用于记录历史阶段决策。

## 1. 作品目标

本项目希望构建一个可持续维护的南京大学仙林校区导航与日程服务，核心目标包括：

1. 使用校园地图图片和人工标注路网，为重要建筑及户外设施提供合理的步行路径规划。
2. 建立两层地图模型：
   - 室外层：以大型建筑和重要户外设施作为可搜索地点，规划地点之间的路线。
   - 室内层：未来以建筑或楼层作为独立区域，以入口、教室、厕所等房间为地点进行室内导航。
3. 让登录用户创建一次性或周期性行程，根据最新路网、最晚到达时间和预计步行时间获得出发提醒。
4. 允许用户提交路径纠错申请，由工作人员在地图上审核后安全合并，形成可持续更新的数据闭环。
5. 提供账号、日程、路径申请、备份和运行状态管理能力，并支持本地运行与服务器部署。

## 3. 技术架构

### 3.1 技术选型

- 前端：原生 HTML、CSS、JavaScript，地图渲染使用 Leaflet。
- 后端：FastAPI。
- 数据库：SQLAlchemy 2.0；本地默认使用 SQLite，结构迁移使用 Alembic。
- 认证：Argon2 密码哈希和短期 JWT access token。
- 路径规划：后端和前端基于 `area.json` 路网计算最短路径。
- 部署：Docker Compose + Caddy，或 Windows 本地 Python 启动。

| `/schedule.html` | 个人日程和提醒 | 登录用户 |

## 4. 地图与路网数据规范

每个区域独占一个文件夹：

```text
data/areas/<area-id>/
├─ map.png
└─ area.json
```

`data/areas/index.json` 是区域注册表。运行时代码不得硬编码某个区域文件路径。

### 4.1 数据对象

正式 `area.json` 主要包含：

- `place`：用户可搜索、可选择的地点，包含名称并绑定一个路网节点。
- `node`：路网交叉口、转折点或接入点，不包含标签。
- `edge`：连接两个节点的可通行边，不保存自由折线。

坐标系统固定为 `image-pixel`。地点、节点和边直接使用底图原始图片上的像素坐标，不使用经纬度。道路弯曲时，应在弯折处增加节点并拆分为多条边。

### 4.2 关键约束

- 一张底图只能对应一个区域，标注过程中不得裁剪或缩放底图。
- `image.width` 和 `image.height` 必须与 `map.png` 原始尺寸一致。
- `place.routeNodeId` 必须引用同区域中的有效节点。
- 边的 `from/to` 必须引用有效节点，不能存在重复边或孤立节点。
- 所有正式地点必须相互可达。
- 正式数据是 `area.json`；数据库不复制地图地点和路网。

### 4.3 数据维护流程

1. 准备区域唯一底图并建立区域文件夹。
2. 标注重要地点、路网节点和节点间边。
3. 运行 `npm.cmd run validate:data` 检查格式、引用和连通性。
4. 在网站上实际选择多个起终点检查路线是否穿楼、绕路或断路。
5. 人工确认后，将正式数据提交到仓库。

用户路径反馈不是上传完整 `area.json`，而是提交增量补丁：

- `add_nodes`
- `add_edges`
- `remove_node_ids`
- `remove_edge_ids`

批准时后端会基于最新正式地图重新应用补丁。地点绑定节点不能删除，删除节点时必须同时删除全部关联边；合并后若出现重复、孤立或不可达地点，批准会被阻止。

## 5. 核心业务流程

### 5.1 地图导航

1. 用户打开地图。
2. 从地点列表、搜索结果或地图中选择起点和终点。
3. 系统根据节点和边组成的图计算路线。
4. 地图显示路线、距离和预计步行时间。
5. 登录用户可将该路线带入日程创建页面。

### 5.2 日程与提醒

行程至少包含日期、最晚到达时间、起点和终点，支持：

- 一次性
- 每天
- 每周
- 每月

系统业务时区固定为 `Asia/Shanghai`。后端每次读取行程时使用最新路网重新计算预计时间，并计算：

```text
建议出发时间 = 最晚到达时间 - 预计通行时间
```

当前提醒属于网站内提醒，页面会定时刷新并高亮临近出发、应立即出发或已经迟到的行程。浏览器关闭后不会发送系统通知、短信或邮件。

### 5.3 路径反馈与审核

用户端：

1. 登录后进入“路径反馈”。
2. 在正式地图上添加节点、连接节点、删除错误节点或边。
3. 填写标题和现场说明并提交。
4. 提交只生成申请，不直接修改正式地图。

工作人员端：

1. 在后台打开“路径申请”。
2. 通过地图预览比较原有路网、新增内容、删除内容和地点。
3. 根据现场说明和地图合理性批准或拒绝。
4. 批准时系统重新校验并原子写入正式 `area.json`，同时保留备份和审核记录。

## 6. 权限与安全边界

| 角色 | 权限 |
|---|---|
| `user` | 地图导航、个人日程、提交路径反馈 |
| `staff` | 查看管理信息、审核路径申请、查看备份 |
| `admin` | `staff` 权限及创建系统备份 |

- 公开注册只能创建 `user`，不能自行获得管理权限。
- 仓库不应分发默认管理员账号；管理员角色由部署者在数据库中授权。
- 密码仅保存 Argon2 哈希。
- 浏览器 token 存于 `sessionStorage`。
- 生产部署必须修改 `NJU_JWT_SECRET` 并使用 HTTPS。
- Web 管理后台不会执行 Git 更新、服务重启或环境变量修改。

## 7. 本地安装与启动

### 7.1 环境要求

- Windows、Linux 或 macOS
- Python 3.10 或更高版本
- Node.js 与 npm，仅用于运行项目检查
- 首次安装依赖时需要网络

### 7.2 Windows 一键启动

将完整项目文件夹复制到目标电脑，双击：

```text
一键打开校园地图.exe
```

启动器会检查 Python、安装缺失依赖、执行数据库迁移、启动 `http://localhost:8000/` 并打开浏览器。启动失败时查看：

```text
%LOCALAPPDATA%\nju-campus-map\launcher.log
```

`.exe` 是项目启动入口，不内置 Python；目标电脑仍需安装 Python 3.10 或更高版本。

### 7.3 命令行启动

首次安装：

```powershell
python -m pip install -r backend/requirements-dev.txt
npm.cmd run db:upgrade
```

推荐同源启动：

```powershell
npm.cmd run dev
```

访问：

- 网站：`http://localhost:8000/`
- API 文档：`http://localhost:8000/docs`
- 健康检查：`http://localhost:8000/api/v1/health`
- 就绪检查：`http://localhost:8000/api/v1/health/ready`

前后端分离开发时，分别运行：

```powershell
.\scripts\start-backend.ps1
.\scripts\start-frontend.ps1
```

此时前端为 `http://localhost:8080/`，API 为 `http://localhost:8000/`。

## 8. 使用指南

### 8.1 游客

1. 打开 `/map.html`。
2. 选择起点和终点。
3. 查看路线、距离和预计步行时间。

游客只能使用地图导航，无法查看个人日程。

### 8.2 普通用户

1. 在首页注册并登录。
2. 登录后进入地图；首次空账号会生成示例日程用于体验。
3. 在地图规划路线后，可用该路线创建日程。
4. 在“我的日程”查看当天提醒或管理周期行程。
5. 发现路网错误时进入“路径反馈”提交修改申请。

### 8.3 工作人员和管理员

1. 使用部署者已授权的 `staff/admin` 账号登录。
2. 进入 `/admin/index.html`。
3. 分页查看用户状态、日程和路径申请。
4. 审核路径申请前先打开地图预览。
5. 管理员可在系统运维页创建地图和数据库备份。

仓库不预置高权限账号。仅在本地评测环境中，可先注册用户名为 `evaluator` 的普通账号，再由部署者在项目根目录执行以下命令授权：

```powershell
@'
from sqlalchemy import select
from backend.app.database import SessionLocal
from backend.app.user_model import User

with SessionLocal() as db:
    user = db.scalar(select(User).where(User.username == "evaluator"))
    if not user:
        raise SystemExit("请先在网页注册 evaluator 账号")
    user.role = "admin"
    db.commit()
    print("evaluator 已授权为 admin")
'@ | python -
```

生产环境应由受控的运维流程授权，不应开放公共角色修改接口。

## 9. 服务器部署

推荐使用安装了 Docker 的 Linux 服务器。

1. 将域名的 `A/AAAA` 记录指向服务器公网地址。
2. 将 `.env.production.example` 复制为 `.env.production`。
3. 设置至少 32 位强随机 `NJU_JWT_SECRET`、真实 `NJU_DOMAIN` 和 `NJU_ALLOWED_ORIGINS`。
4. 开放服务器 `80` 和 `443` 端口。
5. 运行：

```bash
docker compose up -d --build
```

Caddy 会自动申请和续期 HTTPS 证书，并把请求转发至 FastAPI。SQLite 数据库和备份存储在 Docker volume `app-runtime` 中。

生产建议：

- 单实例、小规模使用可保留 SQLite；多实例应更换 PostgreSQL。
- 定期创建备份，并将备份复制到服务器外。
- 监控 `/api/v1/health/ready`。
- 使用强随机密钥、HTTPS 和受保护的 `master` 分支。
- 服务器更新使用 `scripts/update-from-github.ps1`，仅允许干净 `master` 分支 fast-forward 更新。

## 10. 测试与验收

完整检查命令：

```powershell
npm.cmd run check
```

该命令依次检查：

- 文本文件 UTF-8 编码
- JavaScript 语法
- 地图数据结构和连通性
- 后端接口与业务测试

单独验证地图数据：

```powershell
npm.cmd run validate:data
```

评测时建议依次检查：

1. 游客地图导航。
2. 注册、登录和个人日程。
3. 路径反馈提交。
4. 管理后台地图预览与审核。
5. 运维状态、备份和健康接口。
6. `npm.cmd run check` 是否通过。

## 12. 已知限制与后续方向

- 当前主要完成室外层，室内导航尚未建设。
- 路线质量高度依赖人工路网标注和现场核验。
- 当前预计时间采用固定步行速度，未考虑拥堵、坡度、天气和道路临时封闭。
- 网站内提醒要求页面处于打开状态，尚无系统推送。
- 尚未提供公开的管理员角色授权界面、邮箱验证和密码找回。
- 路径申请支持增量安全合并，但仍需工作人员结合实际情况审核。
