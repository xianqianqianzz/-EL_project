# 四人小组人机协作操作指南

本指南面向没有 GIS/OpenCV 经验的小组成员。主流程使用项目自带网页工具、JSON 校验脚本和人工审核。

## A 组：地图渲染与标注工具

### 目标

让 D 组能在网页里添加节点并连边，让正式地图按图结构显示路径。

### 每次开发前

1. 拉取最新代码。
2. 启动本地服务：

   ```bash
   python -m http.server 8080
   ```

3. 打开：

   ```text
   http://localhost:8080/tools/path-editor.html
   ```

### 维护标注工具的步骤

1. 确认图片能加载。
2. 确认点击图片能生成节点。
3. 确认“连边 edge”模式能通过点击两个已有节点生成边。
4. 确认导出的 JSON 包含：
   - `places`
   - `nodes`
   - `edges`
   - `links`
   - `reviewStatus`
5. 确认审核后的导出结果能直接替换对应区域的 `area.json`。
6. 如果 D 组觉得操作困难，优先优化按钮、提示和撤销功能，不急着加复杂功能。

### 正式地图渲染职责

1. 室外路径应按 A* 返回的连续节点坐标画线。
2. 室内路径应按楼层图片坐标或已转换坐标画线。
3. 弯路必须由多个节点和多条边表示。
4. 路线分段时保留室外段、室内段、跨层段。
5. 新区域优先从 `data/areas/<area-id>/area.json` 加载，不再默认扫描混合图片目录。

### 交付物

- `tools/path-editor.html`
- 路线节点连线渲染逻辑
- 简短说明：标注工具有哪些已知限制

## B 组：路径规划与校验

### 目标

让 A* 只沿可通行路径网络行走，并让数据错误尽早暴露。

### 路径数据检查步骤

1. 运行：

   ```bash
   npm.cmd run validate:data
   ```

2. 如果出现“致命问题”，先不要合并。
3. 根据错误定位：
   - 缺少节点：让 D 组补节点或修 ID
   - 边端点不存在：让 D 组修正节点或重新连边
   - 目标不可达：检查 `routeNodeId` 是否接入路网
   - 跨层不可达：检查楼梯/电梯节点是否连接

### 建图规则

1. Graph 使用 `edges[].from/to` 作为拓扑。
2. 路线距离使用边两端节点的像素距离乘 `image.metersPerPixel`。
3. A* 返回节点列表，`PathRenderer` 按节点顺序绘制路线。
4. `walkable: false` 的边不参与寻路。

### 推荐测试路线

- 南门 → 逸夫楼入口
- 南门 → 逸夫楼 A301
- 逸夫楼入口 → 3F 卫生间
- 图书馆 1F → 图书馆 5F
- 图书馆 → 仙 I 教学楼

### 交付物

- Graph 构建和路径规划逻辑
- `scripts/validate-data.js`
- 示例路线测试结果

## C 组：UI 与用户反馈

### 目标

让普通用户不需要理解节点 ID，也能看懂路线和错误提示。

### 搜索交互规则

1. 起点和终点输入框旁必须提供“选择起点”和“选择终点”按钮。
2. 进入地图选择状态后显示全部可选路网节点；完成选择或取消后隐藏节点。
3. 搜索结果分成：
   - 建筑
   - 室外目标
   - 室内房间
   - 室内设施
4. 显示名称用 `label/name`，不要显示 `routeNodeId`。
5. 传给 `app.js` 的搜索项必须包含 `routeNodeId`。

### 路线说明规则

路线说明应写成人话：

- “从南门进入校园”
- “沿中大路向北步行”
- “到达逸夫楼入口”
- “进入 1F，乘电梯到 3F”
- “沿 3F 中央走廊到 A301”

不要写：

- `road-s1 -> road-s2`
- `yifu-3f-corridor-center`

### 标注工具 UI 协助

C 组可以帮助 A 组优化：

- 按钮文案
- 当前模式提示
- 错误提示
- 导出前检查提示
- 操作说明面板

### 交付物

- 搜索结果分类显示
- 路线步骤展示
- 标注工具操作提示

## D 组：数据整理、标注与审核

### 目标

把图片转成可靠路径数据，并记录来源和审核状态。

### 图片整理步骤

1. 为每个区域建立独立文件夹：

   ```text
   data/areas/outdoor-xianlin/
   ```

2. 将该区域唯一底图命名为：
   - `map.png`
3. 将正式数据命名为：
   - `area.json`
4. 阶段导出放在项目外或单独工作目录，正式区域目录中只保留一个 `area.json`。
5. 在记录表中写明来源、楼层、是否清晰。

### 标注步骤

1. 打开 `tools/path-editor.html`。
2. 填写地图名称、建筑 ID、楼层。
3. 导入图片。
4. 需要搜索的目标使用“地点 place”，例如入口、房间、厕所；工具会自动生成对应路网节点。
5. 路口、走廊转角等只用于寻路的位置使用“路网节点 node”。
6. 使用“连边 edge”连接可直接通行的相邻节点：
   - 沿道路中心线
   - 沿走廊中心线
   - 房间门接到最近走廊
   - 道路弯曲处先添加节点，再逐段连接
   - 先点击第一个节点，确认绿色“1”高亮正确，再点击第二个节点
7. 不再选择地点、节点或边的子类型；只有地点需要填写标签。
8. 不确定的线设置为 `needs-review`。
9. 导出 JSON。
10. 放到对应区域文件夹下。
11. 交给 B 组校验；审核通过后直接替换正式 `area.json`。

### 审核步骤

每张图至少做 3 条路线：

- 入口 → 近目标
- 入口 → 远目标
- 涉及楼梯/电梯或跨区域的目标

如果发现路线不合理：

1. 找出穿墙或绕路位置。
2. 回到标注工具补节点或调整边。
3. 重新导出。
4. 再跑校验。

### 交付物

- 原始图片
- 标注 JSON
- 审核记录
- 无法确认的问题清单

## 每周协作节奏

### 第 1 天

D 组整理图片，A 组确认标注工具可用。

### 第 2-3 天

D 组标注第一条演示路径，B 组加校验，C 组准备搜索和路线说明。

### 第 4-5 天

四人一起联调“南门 → 逸夫楼 → A301”。

### 第 6-7 天

补图书馆或第二条室外路线，整理文档和演示截图。

## 遇到问题时怎么问 AI

推荐提问格式：

```text
这是我导出的标注 JSON，校验报错为 xxx。
请帮我判断是 ID 错误、断边，还是 routeNodeId 未接入路网。
不要直接改数据，先说明原因和修改方案。
```

对于看不清的图片，不要让 AI 猜。标记为 `needs-review`，找更清晰图片或现场确认。
