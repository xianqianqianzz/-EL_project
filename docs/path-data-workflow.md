# 图片到可行路径数据的人机合作流程

本流程的目标是把地图图片和楼层平面图转成 A* 能使用的路径网络。项目不要求四名组员都会专业 GIS 软件，主流程使用项目自带网页标注工具完成。

## 1. 总体流程

1. D 组按区域整理图片源。
2. 每个区域建立独立文件夹，例如 `data/areas/outdoor-xianlin/`。
3. 区域文件夹中只保留一张用于标注的 `map.png`。
4. A 组维护 `tools/path-editor.html` 标注工具。
5. D 组用标注工具描出可通行路径。
6. 标注工具直接导出可运行的区域 `area.json`。
7. B 组运行校验脚本，检查断点、重复 ID、不可达目标。
8. C 组把错误提示整理成人能理解的操作反馈。
9. A/B/C/D 联合抽测示例路线。
10. 审核通过的数据替换对应区域的正式 `area.json`。

## 2. 图片准备

D 组先把地图图片整理到对应区域文件夹。已经落地的样板是：

```text
data/areas/
└── outdoor-xianlin/
    ├── map.png
    └── area.json
```

命名规则：

- 区域文件夹：`outdoor-xianlin`、`indoor-yifu-1f`、`indoor-library-2f`
- 区域底图：统一命名为 `map.png`
- 正式区域数据：统一命名为 `area.json`
- 标注工具阶段导出：放在项目外或单独工作目录，不放入正式区域文件夹
- 不确定来源或清晰度不足的图片不要直接进入正式区域文件夹，可先放在 `assets/source-maps/pending/`。

每张图需要记录：

- 图片来源
- 对应建筑/楼层
- 是否最新
- 是否有方向问题
- 是否有遮挡、水印、模糊区域

## 3. 标注原则

只标“人能走的线”，不要标装饰性边界。

室外应标：

- 道路中心线
- 广场/开放空间中实际推荐走线
- 校门到路网的连接线
- 建筑入口到最近道路节点的连接线
- 桥、坡道、可通行连廊

室外不应标：

- 建筑外轮廓
- 草坪边界
- 河流、湖岸、装饰线
- 图片文字标注框

室内应标：

- 走廊中心线
- 房间门口到走廊的短连接线
- 楼梯、电梯点
- 入口大厅到主走廊
- 厕所、服务台等设施到最近走廊的连接线

室内不应标：

- 房间中心到房间中心的直线
- 穿墙线
- 穿过桌椅、书架、设备间的线
- 没有门的房间连接线

## 4. 标注工具使用步骤

1. 启动本地服务：

   ```bash
   python -m http.server 8080
   ```

2. 打开：

   ```text
   http://localhost:8080/tools/path-editor.html
   ```

3. 填写 `mapName`、`buildingId`、`floor`。
4. 选择图片文件。
5. 如需继续上次工作，导入已有 `area.json`。
6. 需要被搜索或点击的目标选择“地点 place”，填写标签后点击图片；若该地点需要进入室内地图，同时填写与室内区域索引项一致的“关联建筑 ID”。
7. 只用于路线转折或分叉的位置选择“路网节点 node”，直接点击图片；节点由工具自动编号，不需要标签。
8. 道路有弯曲时，先在每个明显转折处添加节点。
9. 选择“连边 edge”，依次点击两个已有节点；第一个节点会被高亮，点击第二个节点后立即生成边。
10. 沿道路顺序重复连边，直到整条路线连通。不要跨过转弯直接连接远处节点。
11. 对不确定的边设置 `needs-review`。
12. 导出 JSON。
13. 把导出的 JSON 放在工作目录审核，不要在正式区域文件夹中保留第二份数据文件。
14. 审核通过后直接替换正式 `area.json`，不再进行经纬度转换。

## 5. AI 可以协助的地方

AI 适合做：

- 检查标注工具直接导出的 `area.json`
- 检查孤立节点、重复 ID、无效端点和边缺失
- 根据错误信息提出修复建议
- 把标注者写的中文备注整理成 `label/reviewNotes`
- 帮忙生成路线验收清单

AI 不应直接决定：

- 某条路是否真的可走
- 房间门到底在哪
- 楼梯/电梯是否跨层连通
- 模糊图片中看不清的结构

这些必须由人工审核。

## 6. 室外数据审核

每一片室外路网至少检查：

- 校门到附近建筑是否合理
- 建筑到建筑是否沿道路走
- 室外目标是否吸附到最近路网节点
- 路线是否绕过明显不可通行区域

建议验收路线：

- 南门 → 逸夫楼入口
- 南门 → 图书馆入口
- 图书馆 → 仙 I 教学楼
- 地铁站 → 一组团食堂

## 7. 室内数据审核

每栋楼每层至少检查：

- 入口 → 近房间
- 入口 → 远房间
- 入口 → 卫生间
- 同层两个房间之间
- 跨楼层房间之间

室内路径要特别检查：

- 是否从房间门进入走廊
- 是否误穿墙
- 楼梯/电梯是否连到同一个垂直交通系统
- 4F/5F 等特殊不连通区域是否被错误连接

## 8. 合并前检查

每次提交路径数据前运行：

```bash
npm.cmd run validate:data
```

如果脚本报告致命问题，不能合并。  
脚本报告致命问题时不能合并；`draft` 边应在主演示路线验收后改为 `reviewed`。

当前校验入口已经以 `data/areas/index.json` 的默认室外区域为准。也就是说，`outdoor-xianlin` 的 `area.json` 是室外演示数据的主来源。

## 9. 最小可交付范围

第一阶段不追求全校完整路径，先做：

- 室外：南门、图书馆、逸夫楼、仙 I、仙 II 之间的高质量路径。
- 室内：逸夫楼 1F-3F，入口到 A301、卫生间、计算机实验室。
- 演示路线：南门 → 逸夫楼入口 → 3F A301。

这条路线做扎实，比全校粗糙连线更有说服力。
